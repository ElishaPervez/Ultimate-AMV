Scene detection & preview caching:
where we are, what to consider, what to do

A side-by-side of our stack (TransNetV2 neural detection + re-encode preview cache) against AMVerge's (keyframe/GOP split + lossless pre-cut files), the tradeoffs of every alternative, and a recommendation for the two asks: a light "scene detection" mode for weak machines, and killing the cache for a faster, more responsive preview path.

Our detector: TransNetV2 (GPU) · CPU mean-diff fallback Their detector: keyframe timestamps (demux-only) Source: github.com/crptk/AMVerge (GPL-3.0)

◆ The one thing that ties both asks together

Our preview cache exists for exactly one reason: our cut points are not on keyframes, so producing a clean clip start requires a decode + re-encode. AMVerge has no cache because its cuts are keyframes, so a lossless stream-copy is enough. That means the "light scene-detection mode" and the "no-cache fast mode" are not two features — they're the same lever: keyframe alignment. Add keyframe-mode detection and the cheap-preview path falls out of it for free.

1

Scene detection

Neural shot boundaries vs. cutting at the encoder's keyframes.

Ours · current

TransNetV2 neural detection

  • What: a neural net finds true content/shot boundaries. GPU path decodes via NVDEC/nelux (software-decode fallback) and runs TransNetV2 (sigmoid 0.5, batch 100 / overlap 50). CPU path is a content-aware mean pixel-difference scorer (threshold 27) — cruder than the net, but far from useless.
  • Heads-up: scenedetect (PySceneDetect) is in our requirements + dependency map but isn't actually imported in the active path — the CPU detector is the mean-diff scorer above, not PySceneDetect. Wiring it up is a free quality win waiting to happen.
  • Speed: slow — decode + inference. GPU helps a lot; CPU is the painful case.
  • Quality: high. Real cuts, content-aware.
  • Output: boundary list → ClipScene; previews rendered lazily (no physical cut).
  • Deps: torch, transnetv2, CUDA. Heavy.
  • File: backend/clip_cli.py → extract()
AMVerge · shipping

Keyframe / GOP split

  • What: demux packets with PyAV, collect every packet.is_keyframe timestamp — no decoding. A pathology guard falls back to keyframe-only decode if packet flags look broken.
  • Speed: near-instant (sub-second) — it just reads container metadata.
  • Quality: coarse. It cuts at every I-frame, so it catches real cuts plus periodic GOP keyframes → over-segments.
  • Cleanup: adjacent-thumbnail similarity (8×8 pool + cosine, dissim<0.10) emits merge hints; user merges false cuts.
  • Deps: PyAV only. Their old content-aware detector (Canny+cosine) was deprecated — "accurate, but too slow."
  • File: backend/utils/keyframes.py

The gap that matters: our weak-hardware story today is the CPU mean-diff path — lower-left: only moderate quality (a content-aware pixel delta — decent on hard cuts, weak on fades / fast motion) and slow, because it still decodes every frame. PySceneDetect, already installed but unwired, would sit higher on the quality axis at similar speed.

Keyframe mode lands far right (instant) at middling quality. With a coarsening/merge pass it climbs toward TransNetV2's quality while staying fast — a strictly better corner than where weak machines sit now.

Options on the table

A

Bare keyframe cuts thin

Ship the raw AMVerge detector, lean on the existing scene UI for manual merging.

Pros
  • Smallest change, fastest to land
  • Sub-second on weak HW
Cons
  • Over-segmentation reads as "broken" on a lot of anime
  • Pushes cleanup onto the user
B

Keyframe + min-scene coarsening recommended v1

Same detector, but merge keyframes closer than N seconds — a near-free extension of the min_clip_seconds we already have. No extra decode.

Pros
  • Kills the worst of the over-seg at zero cost
  • Still instant, still no new deps
  • Makes the mode actually pleasant to use
Cons
  • A fixed time-gap heuristic isn't content-aware
C

Keyframe + similarity auto-merge later

Port AMVerge's cs_scenedetect adjacent-thumbnail cosine check to auto-merge near-identical neighbours.

Pros
  • Best quality of the fast options
  • Reuses thumbnails we already make
Cons
  • Needs a thumb decode per scene — erodes the "instant" win somewhat

◆ What "keyframe + similarity auto-merge" actually is

The problem it solves: raw keyframe detection cuts at every I-frame, so one real scene often gets sliced into several near-identical clips. cs_scenedetect is AMVerge's tiny cleanup pass (file backend/utils/cs_scenedetect.py; "cs" = cosine similarity) that stitches those back together.

How it works — dead simple, no neural net: for each pair of adjacent scenes it takes their thumbnails, shrinks them to an 8×8 blur (average-pool), and measures how similar they are (cosine similarity). If two neighbours look almost the same (dissimilarity < 0.10), the split between them was almost certainly a false cut from a periodic keyframe — so it merges them.

Why that's "fast + good + light" at once: keyframe detection gives the instant, no-decode cut list (fast + light); the similarity pass throws away the false cuts to recover something close to content-aware quality (good) — at the cost of decoding one tiny thumbnail per scene, not every frame. It's a cheap stand-in for what TransNetV2 does with a network: "are these two shots actually different?" — answered with 64 pixels and a dot-product instead of a GPU.

▶ Verdict · detection

Add it, default off, as option B. It's a genuinely better weak-machine path than today's CPU mean-diff (which is slow and bad), it adds no new dependency (we already bundle ffprobe — no PyAV needed; read keyframe flags via ffprobe -show_packets), and it returns the identical ClipScene shape so the whole downstream is untouched. Gate it on the cheap coarsening — shipping it bare (A) is the one way to make it feel broken. Keep C as a fast-follow if over-seg complaints persist.

Preview caching

Why we re-encode-and-cache, why AMVerge doesn't, and whether "no cache" is real.

Ours · current

Re-encode preview cache

  • What: each scene preview is a freshly re-encoded mp4 (scene_clips/*.mp4) via dual--ss (coarse before -i for a fast keyframe seek, precise after -i for a clean cut), NVENC p1 @ 720p.
  • Why cache: the cut sits at a non-keyframe boundary, so a clean start requires decode + re-encode. We cache the result so re-selecting is instant.
  • We tried -c copy and rejected it: accurate-seek copy discards leading I-frames mid-GOP → mp4s WebView2 can't decode. Documented dead-end.
  • Already optimised: the encode itself is ~10 ms for a 2 s clip; old cost was the demux walk (fixed by dual--ss) and cold-start (fixed by startup warmup).
  • File: src-tauri/src/clips.rs → render_scene_clip_job
AMVerge · shipping

No cache — because it never seeks

  • What: at split time it writes one standalone .mp4 per scene via ffmpeg -c copy -f segment — lossless stream-copy, no encoder.
  • Playback: the <video> points at the whole small file and plays from t=0. It never seeks into the long source. No bleed, no decode-and-discard.
  • Why it's lossless: cuts are at keyframes, so every segment starts on an I-frame and is independently decodable — the exact failure that killed our -c copy attempt can't happen here.
  • Still has a proxy layer: HEVC / AC3 / Opus sources WebView2 can't play get a transcoded proxy (ensure_preview_proxy). So it's "cache-light," not "cache-free."
  • File: backend/app.py → trim_scenes_at_keyframes

▶ Verdict · caching

"Get rid of the cache" only works in keyframe mode. With TransNetV2 cuts, the re-encode is unavoidable for a clean start and is already near-optimal — a "faster format" optimises the cheap part. Pure no-cache/seek-the-source is a trap for a grid (N elements seeking into one huge file, plus the bleed we already fixed). The real win is: in keyframe mode, build previews via lossless stream-copy instead of NVENC — the cache becomes ~free to build, and we already have the plumbing (clips.rs does keyframe-snapped stream-copy today for lossless merge/export). Fall back to the re-encode proxy only for HEVC / exotic-audio sources.

3

Flawless accuracy and a light grid — without the per-clip cache

The real question: keep TransNetV2's exact cuts, but stop re-encoding and baking an asset per scene.

◆ The codec law (why there's no totally-free lunch)

To display an arbitrary frame of an inter-coded video, a decoder must start from the previous keyframe and roll forward — that's just how H.264/HEVC work. So a precise start at a non-keyframe boundary has only two escapes: (a) re-encode that region into a new clip (what our cache does, ×N), or (b) decode from the nearest keyframe and simply start the display at the in-point — the pre-roll frames are decoded but never shown. Option (b) is free, and it's the one we're not using.

The shift: accuracy is a display concern, not a cutting concern

(a) What we do now — re-encode per clip
I-frame
scene start

We re-encode [scene start → end] into a fresh mp4 so it physically starts at the cut. Clean — but it's an encode and a cached file for every scene.

(b) Offset playback — no new file
I-frame
start display here

Point <video> at an existing source, set currentTime = sceneStart, loop [start,end]. The decoder eats the pre-roll silently; the user only ever sees the green range. Frame-accurate, zero re-encode.

So what does the grid point at?

1

Friendly source → the original itself zero materialization

If the import is already H.264 + AAC at a sane resolution, every grid tile plays the original file with in/out offsets. No proxy, no cache, no baked WebP. The existing IntersectionObserver + stagger queue already cap how many tiles decode at once.

2

Unfriendly source → one shared low-res proxy one pass, not N

HEVC / AV1 / AC3 / Opus / 4K can't play inline or are too heavy to decode ×N. Build a single short-GOP, low-res proxy of the whole episode once (NVENC, fast), then every tile offset-plays that. Short GOP → seeks land precisely & cheaply. One small file replaces hundreds of baked clips.

3

Deep option → MSE byte-range feeding elegant, complex

For friendly codecs you can append just the relevant GOP byte-ranges of the original to a MediaSource with a timestamp offset — precise preview, zero transcode and zero disk. Most elegant, but fragmented-mp4 only and a lot more plumbing; HEVC / odd-audio still fall back to a proxy.

⚠ Honest caveats

  • Not literally zero work for unfriendly sources. They need one whole-episode proxy pass — but it's one NVENC pass at low res (thousands of fps), replacing N per-clip encodes, and it's a single small file rather than a folder of hundreds.
  • Upfront latency vs today's progressive thumbnails. Our current grid shows the first clips fast as WebPs stream in; a whole-episode proxy must exist before tiles go live. Mitigate by building it segmented in scene order (early tiles light up first) and keeping the cheap poster thumbnail visible until then.
  • Looping needs a little JS. Playing [in,out] and looping back is a requestVideoFrameCallback handler per tile — more than AMVerge's "play a whole file," far less than maintaining a render cache.
  • All tiles share one decode source. Low-res keeps it cheap; if the proxy fails, the grid falls back to thumbnails.
  • Export is unchanged. This is a preview-only path; exact export still cuts from the original as today.
▶ Verdict · flawless + light

This is the objectively-better thing to prototype if the goal is "TransNetV2 accuracy + a featherweight grid + no cache folder." The unlock is conceptual: stop materialising clips to get accuracy — get accuracy from the playback in-point instead, and share one decode source. It's orthogonal to detection mode (works with our existing precise cuts). No zero-cost lunch — the codec law means unfriendly sources still need one proxy pass — but it converts "re-encode + cache every scene" into "transcode once, or not at all."

The convergence

Why these are one feature, not two.

Ask #1 — light detection

Fast, GPU-free scene detection for weak machines.

⟶
◆ Keyframe alignment

Cut points are I-frames.

Ask #2 — kill the cache

A faster, more responsive preview path.

⟶
◆ same lever

Unlocks lossless copy & whole-file play.

Detection win

Sub-second, no torch/CUDA, no new dependency — strictly better than today's CPU fallback.

Caching win

Keyframe starts → stream-copy previews → cache is ~free to build, no re-encode, no bleed.

Corollary worth stating loudly: a naïve "no-cache mode" bolted onto TransNetV2 — one that seeks the source and trusts the cut to be clean — would fight the boundary math and reintroduce the leakage we already fixed. The cheap-via-keyframe path is a property of keyframe mode. If you want a cache-light grid that keeps TransNetV2's flawless cuts, that's the separate offset-playback lever in §3 — it sidesteps the leakage by not relying on a clean physical cut at all.

What we should do

Phased, lowest-risk-first. Each phase ships independently.

0
no code

Keep TransNetV2 + its cache as the default

It's the quality path and its cache is already optimised. Nothing below replaces it — everything is additive and default-off.

1
~1 day

Add keyframe detection mode (default off) + min-scene coarsening

The weak-machine detector. Option B from §1.

  • New extract_keyframe() branch in clip_cli.py's run_detection() — reads keyframe flags via ffprobe -show_packets (no PyAV, no new dep).
  • Reuse our min_clip_seconds merge to coarsen sub-N-second gaps.
  • Returns the identical (scenes, cuts) shape → grid, modal, padding all untouched.
  • Toggle in AppConfig + DEFAULT_CONFIG, surfaced near clip_extraction_mode, threaded through clips.rs::clip_extract.
2
~1–2 days

Cache-light previews in keyframe mode

The "faster / no heavy cache" payoff — falls straight out of phase 1.

  • When cuts are keyframe-aligned, materialise previews via lossless stream-copy, reusing the existing keyframe-snap merge plumbing in clips.rs instead of NVENC re-encode.
  • Keep the re-encode proxy as fallback for HEVC / AC3 / Opus (WebView2 can't decode them) — same approach AMVerge uses.
  • Optional: AMVerge-style whole-file-per-scene if we ever want zero per-click work (costs disk).
3
opt · ~1 day

Similarity auto-merge (only if over-seg still bites)

Option C from §1 — quality polish for keyframe mode.

  • Port cs_scenedetect's adjacent-thumbnail 8×8 cosine check to auto-merge near-identical neighbours.
  • Reuses thumbnails we already generate; turns "coarse" into "near-TransNet on web-rips."
▶ The short version

Build Phase 1 + 2 together on this branch: one feature — keyframe mode — that simultaneously answers "fast detection for weak machines" and "faster, cache-light previews." Treat Phase 3 as a fast-follow gated on real over-segmentation feedback. Separately, if you want a lighter grid without giving up TransNetV2's flawless cuts, spike the offset-playback / single-proxy approach in §3 — and consider the quick win of finally wiring the already-installed PySceneDetect as the CPU detector.

⚠ Caveats & honest limits

  • Keyframe mode over-segments by nature. Encoders place I-frames at scene changes and at fixed GOP intervals. Quality is hostage to the encode; coarsening/merge is mandatory, not optional, to make it pleasant.
  • Cache-light ≠ cache-free. WebView2 can't decode HEVC / AC3 / Opus / some HEVC profiles. Those still need a transcoded proxy — exactly the layer AMVerge keeps. The win is real for the common H.264 + AAC case.
  • No-cache/seek-the-source is a trap for the grid. Many <video> elements seeking into one large source is slower and reintroduces bleed. AMVerge deliberately pre-splits instead.
  • "Faster cache format" is low-leverage. Our re-encode is already ~10 ms/clip; the prior costs (demux walk, cold-start) are already fixed. Optimising the format optimises the cheap part.
  • Export precision is separate. Snapping a cut to a keyframe is fine for preview but shifts the export boundary by up to a GOP — keep export on the precise sourceStart/sourceEnd path as today.
  • GPL-3.0. AMVerge is GPLv3; lifting code verbatim carries license obligations. Re-implementing the (simple) keyframe idea against our own ffprobe plumbing sidesteps that and avoids the PyAV dependency entirely.
